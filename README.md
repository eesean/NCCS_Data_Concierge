# NCCS Data Concierge

A natural-language query assistant for the National Cancer Centre Singapore (NCCS) dataset. Users ask questions in plain English; the system generates, validates, and executes DuckDB SQL against the NCCS parquet files and returns structured results with a plain-language summary.

---

## Architecture overview

```
User (Streamlit UI)
    │  HTTP SSE stream
    ▼
FastAPI backend (api.py)
    │
    ▼
Agent.py  ←  deterministic hybrid agent / query pipeline
    │
    ├─ Step 1: get_schema_context()   ← always, hardcoded
    ├─ Step 2: get_sql_template()     ← always, hardcoded
    ├─ Step 3: LLM decides            ← get_cancer_info() if cancer query
    ├─ Step 4: LLM generates SQL      ← focused prompt, no tools
    ├─ Step 5: validate_sql_query()   ← loop until valid (LLM fixes errors)
    ├─ Step 6: get_data()             ← execute validated SQL via DuckDB
    ├─ Step 7: LLM summarises results ← 1-2 sentence plain-language answer
    └─ Step 8: evaluate_live_query()  ← evaluate model performance
    │
    ▼
retrieval/
    llm.py              ← ollama.chat() wrapper (think=False)
    graph/
        tool/
            vectorRag.py       ← ChromaDB vector search (schema, cancer ICD-10, SQL templates)
            SQLvalidator.py    ← DuckDB SQL safety + schema validation
            get_data.py        ← DuckDB query execution against parquets
            evaluation_update.py ← live query logging and scoring
        outputParser.py        ← extract data/text from message history
```

## Setup — fresh clone on a new computer

### 1. Clone the repository

```bash
git clone https://github.com/eesean/NCCS_Data_Concierge.git
cd NCCS_Data_Concierge
git checkout local_llm_implementation
```

### 2. Install Ollama and pull the model

Download and install Ollama from [https://ollama.com](https://ollama.com).

Ollama starts automatically as a background service on macOS after installation.
To verify it is running:

```bash
curl http://localhost:11434
# Expected: "Ollama is running"
```

Pull the model used by the agent:

```bash
ollama pull qwen3:8b
```

### 3. Create a Python virtual environment

```bash
python3 -m venv venv
source venv/bin/activate          # macOS / Linux
# venv\Scripts\activate           # Windows
```

### 4. Install dependencies

```bash
pip install -r requirements.txt
pip install ollama                 # Ollama Python client
```

> **Note:** If you encounter an `OSError: could not get source code` on startup (Python 3.9 + PyTorch 2.5+), downgrade transformers:
> ```bash
> pip install "transformers==4.46.3"
> ```
> If pip has SSL issues, add `--trusted-host pypi.org --trusted-host files.pythonhosted.org`.

### 5. Add the parquet data files

Place the following parquet files in `data/` at the **project root** (this matches `SQLvalidator.py`):

```
condition_occurrence.parquet
death.parquet
drug_exposure_cancerdrugs.parquet
measurement_mutation.parquet
person.parquet
procedure_occurrence.parquet
```

These files are **not committed** to the repository (excluded via `.gitignore`).

### 6. Create the `.env` file

Create a `.env` file in the project root:

```env
OLLAMA_MODEL=qwen3:8b
OLLAMA_BASE_URL=http://localhost:11434

PARQUET_KEY_NAME=<your_key_name>
PARQUET_KEY_VALUE=<your_key_value>
```

> `PARQUET_KEY_NAME` and `PARQUET_KEY_VALUE` are encryption credentials for the parquet files. Obtain these from your project administrator.
> `OPENROUTER_API_KEY` is no longer required on this branch.

### 7. Build the ChromaDB vector store

If `retrieval/chroma_db/` is missing (not committed for large repos), regenerate it by running the setup notebook:

```bash
cd retrieval
jupyter notebook exploratory.ipynb
```

Run all cells in the notebook. This embeds the schema columns, ICD-10 cancer reference, and SQL templates into ChromaDB using the `sentence-transformers/all-mpnet-base-v2` model.

### 8. Run the backend (FastAPI)

```bash
# From the project root, with venv activated
uvicorn api:app --host 127.0.0.1 --port 8000
```

Wait for:
```
INFO:     Application startup complete.
INFO:     Uvicorn running on http://127.0.0.1:8000
```

> Do **not** press Ctrl+C during startup — the import chain (torch, sentence-transformers, ChromaDB) takes 30–60 seconds on the first run.

### 9. Run the frontend (Streamlit)

In a **separate terminal** (venv activated):

```bash
streamlit run NCCS_Query_Assistant.py
```

The UI opens at `http://localhost:8501`.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `OSError: could not get source code` on startup | PyTorch 2.5+ + Python 3.9 incompatibility | `pip install "transformers==4.46.3"` |
| `ImportError: Could not import sentence_transformers` | Import interrupted by Ctrl+C during startup | Restart uvicorn and wait for full startup |
| `Read timed out` after 300s | Model in thinking mode | Confirm `think=False` is in `retrieval/llm.py`; restart backend |
| `Error: listen tcp 127.0.0.1:11434: bind: address already in use` | Ollama already running | Normal — Ollama runs as a background service; no action needed |
| WatchFiles constantly reloading venv | uvicorn watching entire project tree | Use `uvicorn api:app --host 127.0.0.1 --port 8000` (no `--reload`) |
| `Could not retrieve data` | `get_data` not reached | Check backend logs for validation errors; SQL failed all retry attempts |
